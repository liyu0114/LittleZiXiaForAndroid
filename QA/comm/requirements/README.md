# 需求会商机制

程序员与质检员协商确定工作需求。

---

## 一、会商流程

```
1. 发起方创建需求文件 → REQ_{时间戳}.md
2. 对方读取并回复意见 → REQ_{时间戳}_REPLY_{时间}.md
3. 双方达成一致 → 需求文件重命名为 REQ_{时间戳}_CONFIRMED.md
4. 开始执行
```

---

## 二、需求文件命名

| 类型 | 格式 | 示例 |
|------|------|------|
| 需求提案 | `REQ_{YYYYMMDD}_{HHMM}.md` | `REQ_20260329_1615.md` |
| 回复意见 | `REQ_{时间}_REPLY_{时间}.md` | `REQ_20260329_1615_REPLY_20260329_1620.md` |
| 已确认 | `REQ_{时间}_CONFIRMED.md` | `REQ_20260329_1615_CONFIRMED.md` |
| 已拒绝 | `REQ_{时间}_REJECTED.md` | `REQ_20260329_1615_REJECTED.md` |

---

## 三、需求文件内容模板

```markdown
# 需求提案

## 时间: YYYY-MM-DD HH:MM
## 发起方: DEV/QA
## 标题: xxx

## 需求描述

(具体需求内容)

## 优先级

P0/P1/P2/P3

## 预计工时

x 小时

## 涉及模块

- 模块1
- 模块2

## 待确认事项

- [ ] 事项1
- [ ] 事项2
```

---

## 四、回复文件内容模板

```markdown
# 需求回复

## 时间: YYYY-MM-DD HH:MM
## 回复方: QA/DEV
## 针对需求: REQ_xxx.md

## 意见

同意 / 有异议

## 补充建议

(如有)

## 修改要求

(如有)

## 结论

✅ 同意执行 / ⚠️ 需修改 / ❌ 拒绝
```

---

## 五、快速确认规则

以下情况**无需会商**，可直接执行：

| 情况 | 说明 |
|------|------|
| Bug 修复 | 已有功能的 bug，直接修 |
| P1 问题 | 质检报告已确认的 P1 问题 |
| 代码优化 | 不改变功能的重构 |
| 文档更新 | README、注释等 |

**需要会商的情况：**

| 情况 | 说明 |
|------|------|
| 新功能 | 新增模块或功能 |
| 架构调整 | 影响整体结构 |
| P2/P3 问题 | 优先级待确认 |
| 争议事项 | 双方意见不一致 |

```
├── requirements/      # 需求会商目录
│   ├── REQ_20260329_1615_CONFIRMED.md
│   └── REQ_20260329_1700.md
└── archive/
```

---

## 七、示例流程

```
[DEV] 发现新需求 → 创建 REQ_20260329_1615.md
[DEV] 飞书通知: 📋 [DEV] 发起需求会商 REQ_20260329_1615.md
[QA] 读取需求 → 创建 REQ_20260329_1615_REPLY_20260329_1620.md
[QA] 飞书通知: 📥 [QA] 已读需求 REQ_20260329_1615.md，已回复
[DEV] 读取回复 → 确认 → 重命名为 REQ_20260329_1615_CONFIRMED.md
[DEV] 飞书通知: ✅ [DEV] 需求确认 REQ_20260329_1615.md，开始执行
[DEV] 开始工作
```

---

## 八、通知格式

| 操作 | 格式 |
|------|------|
| 发起需求 | `📋 [角色] 发起需求会商 文件名` |
| 回复需求 | `📝 [角色] 回复需求 文件名` |
| 确认需求 | `✅ [角色] 需求确认 文件名，开始执行` |
| 拒绝需求 | `❌ [角色] 需求拒绝 文件名` |

---

## 九、响应时限

| 类型 | 响应时限 | 说明 |
|------|----------|------|
| 普通需求 | 24小时内 | 正常工作日 |
| 紧急需求 | 2小时内 | 文件名加 `_URGENT` |

### 紧急需求标识

```markdown
# 文件命名
REQ_20260329_1615_URGENT.md

# 内容标识
## 优先级: P0-URGENT
## 紧急原因: xxx
```

---

## 十、争议解决

当双方意见不一致时：

1. **协商** — 在回复文件中说明分歧点
2. **Liyu 仲裁** — 如无法达成一致，由 Liyu 做最终决定
3. **记录** — 争议和决议记录在文件中

---

## 十一、归档规则

- 已完成的需求文件（CONFIRMED/REJECTED）每月归档到 `archive/`
- 保留最近 3 个月的需求文件在 `requirements/` 目录
