## 一、大模型扩展需求

### 1.1 问题

**当前实现：** 硬编码 7 个提供商
- 通义千问、OpenAI、Claude、GLM、DeepSeek、Ollama、自定义

**局限性：**
- 新模型不断涌现，无法穷尽
- 更新需要修改代码
- 用户无法自定义添加

- 百川

3. 支持导入/导出配置

**优点：**
- 立即可用

## 二、无障碍服务需求

### 2.1 背景

让小紫霞像人一样能够使用手机上已安装的 APP，通过官方合法渠道实现。

### 2.2 技术方案

#### 2.2.1 前台服务 (Foreground Service)

**作用：** 确保 AI 应用在后台持续运行不被系统杀死

**支持的服务类型：**
- microphone - 后台持续监听音频
- mediaProjection - 实时捕获屏幕内容
- phoneCall - 管理通话
- shortService - 短时关键任务（约3分钟）

#### 2.2.2 无障碍服务 (Accessibility Service)

**核心能力：**
1. **感知屏幕内容** - 获取屏幕上所有 UI 元素
2. **模拟用户操作** - 点击、长按、输入、滑动
3. **监听全局事件** - 应用切换、通知弹出

### 2.3 开发计划

| Phase | 任务 | 工时 | 内容 |
|-------|------|------|------|
| 1 | 基础框架 | 4h | 创建 AccessibilityService |
| 2 | 屏幕感知 | 3h | 解析 UI 元素树 |
| 3 | 操作执行 | 3h | 点击、输入、滚动 |
| 4 | 应用知识库 | 4h | 主流应用操作脚本 |

**总工时：** 14小时

### 2.4 用户场景示例

```
用户: "帮我给张三发微信说我要迟到了"
  ↓
小紫霞:
  1. 打开微信
  2. 找到张三的聊天
  3. 输入消息
  4. 发送
  ↓
反馈: "已发送"
```

### 2.5 待确认问题

1. **优先实现哪些应用的操作脚本？**
   - 微信（发消息、朋友圈）
   - 支付宝（付款、转账）
   - 其他？

2. **是否需要屏幕截图能力（mediaProjection）？**
   - 用于 AI 视觉理解屏幕内容
   - 需要额外权限申请

## 三、记忆系统升级需求（R1）

### 3.1 背景

对标 OpenClaw 的 MEMORY.md 机制，实现两层记忆架构。

### 3.2 实现方案

**文件结构：**
```
memory/
├── MEMORY.md          # 长期记忆
├── 2026-03-30.md      # 每日日志
└── 2026-03-29.md      # 昨日日志
```

**加载策略：**
- 启动时加载当日和昨日日志
- 主会话加载 MEMORY.md
- 文件即上下文，持久化能力强
